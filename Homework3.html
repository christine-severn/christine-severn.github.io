Accessibility checklist

Website Page Titles

In the web page markup, the <title> is within the <head> tag. Good page titles are particularly important for orientation, they help people know where they are and move between pages open in their browser tab. The first thing screen readers say when the user goes to a different web page is the page title.

Results: 10 pages on the MPL website were evaluated for clear, unique HTML page titles. Each page has descriptive HTML title values. However, we do suggest that the title values have to be formatted as "Page Title at MPL", for example, "Study Rooms at MPL" so that it's obvious to users using the assistive technologies what website page they are on.

Image Text Alternatives

Image text alternatives ("alt text") (pictures, illustrations, charts, etc.) Text alternatives convey the purpose of an image, including pictures, illustrations, charts, etc. The alt attribute provides an alternative text for an image, if the user for some reason cannot view it. The reason can be a slow connection, an error with the image file, or if the user uses a screen reader.

Results: No photos on the Homepage, Events, and Spaces pages had descriptive alt text. This means that people using assistive technologies would not know what the photo or graphic is communicating. We recommend that all photos and graphics have descriptive alt text.

Text:

Headings, Contrast Ratio ("color contrast") and Resizing Text should not only support visual aesthetics but also support the navigation and visibility of text content.

Results - Headings We tested the About, Events, and Spaces pages and headings tags are not really nested in a heirarchical system. It appears headdings are used more for font functions then a heirarchical system of pages and information. The MPL homepage has 2 H1 tags, there should only be one H1 per page because screen readers expect only one H1 tag.

Results - Contrast Ratio We tested the Homepage, Services, and Events pages and there was low color contrast because the pages are white and the headings are in bright or neutral colors (i.e. orange, blue, teal, etc.). The WAVE color contrast tester did flag these pages as a color contrast issue because of the white background and orange or teal/blue text. Also, the white text on the Homepage header image does not comply with color contrast suggestions as the image header is a carousel and the overlying text does not adequately contrast with the carousel images. Suggestion is to make the text a darker, more contrasting color or to update the image carousel and navigation options so they are not overlapping.

Results - Resizing Text The MPL website testing done with the WAVE text size tester shows failed results for text sizing and resizing. The text for the Homepage does not increase or adjust reeadability for desktop or mobile devices which can cause issues for those with visual disabilities.

Interaction:

Keyboard access and visual focus

Many people cannot use a mouse and rely on a keyboard to interact with web pages. Keyboard focus should follow a logical order through the page elements. Visible keyboard focus could be a border or highlight.

Results: Using only keyboard keys to navigate the MPL website was very difficult. We could not open the sub-pages in the main navigation using our keyboard keys. There was no Skip to Main feature that allows keyboard users to quickly skip to the main content section. Furthermore, Keyboard Focus was not obvious to us when we were using our tab and arrow keys. There was no way to differentiate which links were being tabbed to (i.e. bolding or highlighting the text or section) aside from the main, top navigation links (i.e. Home, About, Events, Spaces, Books & More, Services) changing to a boldened orange text. But this is not acceptable as it is hard to notice when using keyboard navigation functions. We recommend that the web team explore to make the MPL website more keyboard-only friendly.

Forms, labels, and errors (including Search fields):

Users are able to keyboard navigate to the different forms fields. Each form control has a label element with a for attribute that matches the value of the id attribute in the related control. For example:
  • Each form control has a label element with a for attribute that matches the value of the id attribute in the related control. For example:
    <label for="firstname">First Name: </label>
    <input type="text" name="firstname" id="firstname" />
  • Each id is unique within the webpage.
